# TDD Green

실패 중인 테스트를 하나씩 통과시키는 **최소한의 구현 코드**를 작성한다.
"최소한"이란 해당 테스트가 요구하는 동작만 구현하고, 테스트가 검증하지 않는 것은 건드리지 않는다는 뜻이다.

```
실패 테스트 목록
    ↓ 첫 번째 실패 테스트 선택
    ↓ 최소 구현 작성
    ↓ 테스트 실행 → 통과 + 회귀 없음 확인
    ↓ issue-{N}.md 체크박스 업데이트
    ↓ 다음 실패 테스트
전체 통과 → 커버리지 측정 → 결과 요약
```

---

## 핵심 원칙

이 원칙들은 절차보다 우선한다. 절차가 모호할 때 판단 기준이 된다.

**테스트가 유일한 명세다**
테스트 파일이 요구하는 것이 전부다. 테스트가 검증하지 않는 동작은 존재하지 않는 요구사항이므로 구현하지 않는다. 구현 충동이 생기면 "이걸 검증하는 테스트가 있는가?"를 먼저 묻는다.

**테스트 파일은 읽기 전용이다**
테스트를 통과시키기 위해 테스트를 바꾸는 것은 명세를 바꾸는 것이다. 테스트가 어색하거나 틀려 보여도 구현 코드로만 해결한다. 테스트 수정이 필요하다고 판단되면 개발자에게 보고하고 중단한다.

**한 번에 한 시나리오만 진행한다**
여러 테스트를 한꺼번에 통과시키려는 시도는 예상치 못한 회귀를 만든다. 첫 번째 실패 테스트 하나만 선택하고, 통과를 확인한 뒤 다음으로 넘어간다.

**가장 단순한 코드가 정답이다**
통과시키는 방법이 여러 개면 가장 짧고 직관적인 것을 선택한다. 미래의 요구사항을 예측해서 추상화하거나 유연성을 확보하려 하지 않는다. 그 판단은 Refactor 단계의 몫이다.

**불필요한 방어 코드를 넣지 않는다**
테스트가 다루지 않는 null 체크, 타입 가드, 기본값 처리는 지금 구현할 필요가 없다. 방어 코드가 필요한 상황은 해당 케이스를 다루는 테스트가 먼저 존재해야 한다.

**디자인 문서 밖의 스타일을 임의로 추가하지 않는다**
`design-system.md`나 기존 컴포넌트에 명시되지 않은 색상, 간격, 애니메이션은 추가하지 않는다. 테스트가 UI 스타일을 검증하지 않더라도, 코드베이스의 시각적 일관성은 디자인 문서가 결정한다.

**회귀 확인은 매번, 예외 없이 한다**
새 테스트를 통과시킨 뒤 반드시 전체 테스트를 실행한다. 회귀가 발생하면 그것은 구현 방향이 잘못됐다는 신호다. 회귀를 감수하고 진행하거나 기존 테스트를 수정하는 것은 허용되지 않는다.

**피드백 루프는 최대 5회다**
같은 테스트에 대해 5회 수정해도 통과하지 못하면 코드 문제가 아닐 가능성이 높다. 테스트 설계 오류, 시그니처 불일치, 환경 문제일 수 있으므로 개발자에게 보고하고 판단을 맡긴다.

---

## 시작 전: 입력 확인

- **이슈 번호**: `$ARGUMENTS`에서 추출.
- **시나리오 파일**: `docs/features/tag/issue-{N}.md`를 읽는다.
  - 없으면: "`issue-{N}.md`가 없습니다. `/test-scenarios {N}`을 먼저 실행해주세요." 안내 후 중단.

---

## 1단계: 전체 테스트 실행

```bash
npm test
```

실패 테스트 목록을 확인한다.

---

## 2단계: collect 실패 감지

vitest가 보고하는 **Tests 수**와 `issue-{N}.md`의 **체크박스(`- [ ]`) 총 건수**를 비교한다.

불일치하면 import 대상 모듈이 없어서 테스트 파일 전체가 인식되지 않는 것이다.
해당 모듈의 스텁 파일을 생성해 모든 테스트가 인식되도록 한 뒤 1단계로 돌아간다.

스텁 패턴은 `tdd-red` 스킬의 스텁 파일 패턴을 참고한다 (시그니처만, 로직 없음).

---

## 3단계: 디자인 컨텍스트 로드

UI 컴포넌트를 구현할 때는 시각적 스펙을 먼저 파악한다.

1. `docs/design-system.md`가 있으면 읽어서 구현 대상 컴포넌트의 색상·간격·레이아웃·상태별 표현을 확인한다.
2. 없으면 기존 컴포넌트(`src/components/`)에서 유사한 패턴을 찾아 참고한다.
3. `docs/features/tag/issue-{N}.md`에 "디자인 참고" 섹션이 있으면 함께 확인한다.
4. `CLAUDE.md`의 스타일링 컨벤션(Tailwind 인라인 클래스, 조건부 클래스 패턴)을 반드시 따른다.

디자인 문서가 전혀 없으면 이 단계를 건너뛰고 `CLAUDE.md` 컨벤션만 따른다.

---

## 4단계: 첫 번째 실패 테스트 구현

실패 테스트 목록 중 **첫 번째**를 선택해 통과시키는 최소한의 코드만 작성한다.

### 구현 원칙

- 테스트가 `expect(A).toBe(B)`라면 A가 B를 반환하도록만 구현한다.
- 아직 테스트되지 않은 엣지 케이스나 확장 기능은 구현하지 않는다.
- UI 컴포넌트는 테스트가 요구하는 동작을 충족하되, 3단계에서 파악한 디자인 스펙의 Tailwind 클래스와 레이아웃 구조를 함께 적용한다.
- 디자인 스펙에 없는 시각적 요소(애니메이션, 추가 스타일)는 임의로 추가하지 않는다.
- `CLAUDE.md`의 구현 패턴(export 방식, Props 타입 선언, 네이밍 규칙)을 따른다.

---

## 5단계: 테스트 실행 — 피드백 루프

```bash
npm test
```

### 통과한 경우

대상 테스트가 통과하고 기존 테스트에 회귀가 없으면 → 6단계로 진행.

### 실패한 경우

에러 메시지를 읽고 원인을 분석해 코드를 수정한 뒤 이 단계를 반복한다.

**최대 5회 반복 후에도 실패하면** 개발자에게 아래 내용을 보고하고 중단한다:

- 실패 중인 테스트 이름
- 에러 메시지 전문
- 시도한 수정 내역과 각각의 결과

---

## 6단계: 체크박스 업데이트

해당 시나리오를 `docs/features/tag/issue-{N}.md`에서 `- [ ]` → `- [x]`로 변경한다.

---

## 7단계: 다음 테스트로 이동

남은 실패 테스트가 있으면 4단계부터 반복한다.

---

## 8단계: 커버리지 측정

모든 테스트가 통과하면 커버리지를 확인한다.

```bash
npx vitest run --coverage
```

미커버 라인을 확인하고 다음 중 하나로 판단해 개발자에게 보고한다:

- **시나리오에서 빠진 케이스**: "이 라인은 테스트되지 않았습니다. 시나리오 추가를 검토해주세요."
- **도달 불가 코드 또는 방어 코드**: "이 라인은 현재 시나리오 범위 밖입니다."

커버리지를 올리기 위해 테스트를 추가하거나 구현을 바꾸지 않는다.

---

## 9단계: 결과 요약

아래 형식으로 요약한다.

```
테스트 파일                    통과   실패
────────────────────────────────────────
useTagEditor.test.ts             8      0
TagInput.test.tsx                5      0
NoteEditor.test.tsx              8      0
────────────────────────────────────────
합계                            21      0

커버리지 (주요 파일)
─────────────────────────────────
useTagEditor.ts     Stmts 100%
TagInput.tsx        Stmts 100%
NoteEditor.tsx      Stmts  92%  미커버: 45-47 (에러 처리 분기)

미커버 라인 판단: NoteEditor 45-47은 시나리오에서 다루지 않은 방어 코드
```

---

## 제약

- 테스트 파일(`*.test.ts`, `*.test.tsx`)은 읽기만 허용. 수정 금지.
- 테스트가 검증하지 않는 기능 구현 금지.
- 불필요한 추상화(공통 유틸, 헬퍼 함수) 금지 — 테스트가 요구하는 것만 구현.
- `design.md`에 없는 시각적 요소 임의 추가 금지.
- 커버리지를 올리기 위한 테스트 추가 금지.
