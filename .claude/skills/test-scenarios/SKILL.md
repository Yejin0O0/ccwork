# Test Scenarios

GitHub 이슈 번호를 받아 **시그니처 확정 → 테스트 시나리오 도출**을 순서대로 처리한다.
이 스킬의 산출물은 이후 TDD 사이클(Red → Green → Refactor)의 입력이 된다.

```
GitHub Issue + prd.md + 코드베이스
    ↓ 시그니처 도출
    ↓ [GATE] 개발자 승인
    ↓ 시나리오 도출 + AC 대조
    ↓ [GATE] 개발자 승인
docs/features/tag/issue-{N}.md
```

---

## 시작 전: 입력 확인

- **이슈 번호**: `$ARGUMENTS`에서 추출. 없으면 질문한다.
- **참조 문서**: `docs/features/tag/prd.md`를 읽는다 — ADR에 담긴 기술 결정이 시그니처 설계의 근거가 된다.
- **GitHub 이슈**: `gh issue view $ARGUMENTS`로 이슈 제목·설명·AC 목록을 읽는다.

---

## 1단계: 시그니처 도출

이슈가 다루는 파일/함수/컴포넌트를 코드베이스에서 탐색한 뒤 시그니처를 설계한다.

탐색 대상:

- `src/types/note.ts` — 타입 정의 및 확장 필드
- `src/api/notes.ts` — API 함수 패턴 (`res.ok` 체크, 타임스탬프 생성)
- `src/context/NotesContext.tsx` — Context 액션 패턴 (async, `Partial<Note>`)
- `src/hooks/` — 기존 훅 패턴 (있을 경우)
- `src/components/` — Props 인터페이스 패턴 (`on[동작]` 이벤트, `is` 접두사 불리언)

**기존 패턴을 따르는 이유**: 코드베이스 전체 일관성을 유지해야 새 코드가 기존 테스트와 충돌하지 않는다.

도출 항목:

- 함수/훅 시그니처 (이름, 파라미터 타입, 반환 타입)
- 컴포넌트 Props 인터페이스 (이름, 각 prop의 타입과 역할)
- 에러 케이스 (어떤 조건에서 throw하는가)

**구현 코드는 절대 작성하지 않는다.** 시그니처와 타입만 작성한다.

### 시그니처 제시 형식

시그니처를 제시할 때는 두 가지로 나눠서 보여준다.

**[확정]** — 이슈 AC, prd.md ADR, 기존 코드 패턴에 의해 이미 결정된 항목.
개발자 선택이 필요 없으므로 근거만 밝히고 확정 표기한다.

**[결정 포인트]** — 명세에 없거나 해석의 여지가 있어 개발자 판단이 필요한 항목.
대안 2개 이상을 제시하고, 제안 이유와 각 안의 트레이드오프를 설명한다.

각 항목을 아래 형식으로 작성한다:

```
### [확정] 항목명

**근거**: 이 시그니처가 이미 결정된 출처 (이슈 AC-N / prd.md ADR-N / 기존 패턴명)

{시그니처 코드}

---

### [결정 포인트] 항목명

**근거**: 어떤 상황이 이 결정을 필요로 하는가. 명세의 어느 부분이 침묵하고 있는가.

**안 A** (제안):
{코드}

**안 B**:
{코드}

**제안 이유**: 안 A를 제안하는 이유. 안 B 대비 구체적인 장점.
**안 B의 단점**: 왜 안 B를 고르지 않는가.
```

결정 포인트가 없는 시그니처는 단순 나열이 아닌 "모두 확정됨" 한 줄로 요약해도 된다.

결정 포인트가 있는 경우, 모든 시그니처 제시가 끝난 뒤 **결정 포인트 요약**을 아래 형식으로 마지막에 묶어준다.
요약만 보고도 무슨 결정인지 이해할 수 있도록 핵심 내용을 한 줄로 담는다.
개발자가 위로 스크롤하지 않아도 안 A / 안 B의 차이를 바로 파악할 수 있어야 한다.

```
**답변이 필요한 결정 포인트:**

1. [항목명] — {이 결정이 다루는 상황 한 줄 요약}
   - 안 A (제안): {한 줄 설명}
   - 안 B: {한 줄 설명}

2. ...
```

예시:

```
**답변이 필요한 결정 포인트:**

1. Note.tags 필드 타입 — 구버전 노트(tags 없음) 호환을 어디서 처리할지
   - 안 A (제안): 필수 string[] — NoteEditor 한 곳에서만 ?? [] 방어
   - 안 B: 선택 string[]? — 호출부 전체에서 방어 코드 필요

2. Context createNote 시그니처 — tags를 어떤 방식으로 받을지
   - 안 A (제안): (title, content, tags) — 기존 패턴 최소 확장
   - 안 B: ({ title, content, tags }) — 객체로 묶기, updateNote와 스타일 혼재
```

### [GATE] 시그니처 승인 대기

결정 포인트 답변을 포함해 **개발자가 "확정"을 줄 때까지 기다린다.**
승인 전에 시나리오 작성이나 파일 저장으로 넘어가지 않는다.

---

## 2단계: 시그니처 저장

승인된 시그니처를 `docs/features/tag/issue-{N}.md` 상단에 기록한다.

```markdown
# Issue {N}: {이슈 제목}

## 시그니처

### 훅 / 함수

(승인된 시그니처)

### 컴포넌트 Props

(승인된 Props 인터페이스)

### 에러 케이스

(승인된 에러 동작)

---

## 테스트 시나리오

(3단계 완료 후 채움)
```

---

## 3단계: 테스트 시나리오 도출

확정된 시그니처를 기반으로 테스트 시나리오를 도출한다.

### 시나리오 분류

| 분류 | 의미                                             | 예시                                               |
| ---- | ------------------------------------------------ | -------------------------------------------------- |
| 정상 | 올바른 입력 → 기대 출력                          | 유효한 태그 추가 시 tags 배열에 포함               |
| 경계 | 제한값의 끝                                      | 정확히 15자인 태그 추가 허용                       |
| 예외 | 잘못된 입력 또는 실패 상황 → 거부·에러·복구 동작 | 16자 태그 → shaking / API 실패 → 저장 전 상태 유지 |

**예외 시나리오는 이슈 범위와 무관하게 항상 포함한다.**
시그니처의 "에러 케이스" 항목이 곧 예외 시나리오의 출발점이다.
이 이슈에서 구현하지 않는 검증 로직(shake 등)은 "이슈 N에서 다룸"으로 명시하되,
이 이슈에서 다루는 에러(API 실패, 잘못된 인자 등)는 반드시 예외 시나리오로 작성한다.

### 시나리오 형식

```
[분류] 함수명/컴포넌트명 — should {기대 동작} when {조건}
```

예시:

```
[정상] addTag — should append normalized lowercase tag when valid input is given
[경계] addTag — should accept tag when input is exactly 15 characters
[예외] addTag — should trigger shaking and not add when input exceeds 15 characters
[예외] addTag — should trigger shaking and not add when duplicate tag is entered
[정상] removeTag — should remove tag at given index from tags array
[경계] useTagEditor — should return empty tags array when initialTags is empty
```

## 4단계: AC 대조

시나리오를 도출한 뒤 GitHub Issue의 AC 목록과 대조한다.

```bash
gh issue view $ARGUMENTS
```

AC 항목마다 대응하는 시나리오가 있는지 확인한다.
커버되지 않은 AC가 있으면 시나리오를 추가한다.

**대조가 중요한 이유**: AC는 "무엇을 검증해야 하는가"의 합의다. 시나리오가 AC를 빠뜨리면 이후 TDD 사이클에서 해당 동작이 테스트되지 않는다.

대조 결과를 표로 제시한다:

```
| AC | 커버 시나리오 |
|----|--------------|
| AC-1(태그 추가) | [정상] addTag — should append... |
| AC-2(소문자 정규화) | [정상] addTag — should normalize... |
| AC-3(빈 입력 무시) | [정상] addTag — should ignore when input is empty |
```

모든 AC가 커버되면 `docs/features/tag/issue-{N}.md`의 "테스트 시나리오" 섹션을 채운다.

---

## 5단계: 시나리오 저장 및 최종 승인

`docs/features/tag/issue-{N}.md`에 시나리오를 추가한다.

```markdown
## 테스트 시나리오

### 정상

- [정상] ...
- [정상] ...

### 경계

- [경계] ...

### 예외

- [예외] ...

---

## AC 커버리지

| AC  | 커버 시나리오 |
| --- | ------------- |
| ... | ...           |
```

### [GATE] 시나리오 승인 대기

완성된 시나리오와 AC 커버리지 표를 개발자에게 제시하고 **"확정"을 받을 때까지 기다린다.**
승인 후에는 테스트 코드 작성 단계로 넘어갈 준비가 된 것이다.

## 게이트 요약

| 지점     | 확인 내용              | 다음 단계 진입 조건                  |
