# TDD Refactor

테스트 통과 상태를 유지하면서 **코드 구조를 개선**한다.
리팩토링은 동작을 바꾸지 않고 코드를 읽기 좋게 만드는 작업이다.
"이 변경이 테스트를 깨뜨리지 않으면서 코드를 더 명확하게 만드는가?"가 모든 판단의 기준이다.

```
전체 테스트 통과 확인
    ↓ 이슈 변경 파일 목록 추출
    ↓ 점검 기준으로 개선 후보 목록 작성
    ↓ [GATE] 개발자 승인
    ↓ 하나씩 변경 → 테스트 → 통과 확인
    ↓ 실패하면 즉시 롤백
전후 변경 요약
```

---

## 시작 전: 입력 확인

- **이슈 번호**: `$ARGUMENTS`에서 추출. 없으면 질문한다.

---

## 1단계: 컨벤션 파악

`CLAUDE.md`를 읽어 이 프로젝트의 네이밍 규칙, 컴포넌트 패턴, 스타일링 방식을 파악한다.
리팩토링은 코드베이스 전체의 일관성을 높이는 방향이어야 한다. 기존 패턴과 멀어지는 리팩토링은 개선이 아니다.

---

## 2단계: 전체 테스트 통과 확인

```bash
npm test
```

실패 테스트가 있으면 중단하고 개발자에게 알린다. 리팩토링은 반드시 Green 상태에서 시작해야 한다.

---

## 3단계: 리팩토링 대상 파일 식별

이번 이슈에서 변경된 `src/` 파일만 리팩토링 대상으로 삼는다. 범위를 벗어난 파일은 건드리지 않는다.

```bash
git diff main...HEAD --name-only -- src/
```

테스트 파일(`*.test.ts`, `*.test.tsx`)은 목록에서 제외한다.

---

## 4단계: 점검 기준으로 개선 후보 목록 작성

남은 파일 각각을 아래 5가지 기준으로 검토한다. 모든 항목을 강제로 찾으려 하지 않아도 된다. 실제로 읽기 어렵거나 유지보수하기 불편한 코드가 있을 때만 후보로 올린다.

### 점검 기준

**1. 중복 제거**
같은 로직이 두 곳 이상에서 반복되는가? 복붙된 코드 블록, 동일한 조건 분기가 여러 함수에 있는 경우가 해당한다. 단, 우연히 비슷해 보이는 코드를 억지로 합치면 오히려 결합도가 높아진다 — 진짜 동일한 의도인지 먼저 확인한다.

**2. 네이밍 명확성**
변수·함수·컴포넌트 이름이 의도를 드러내는가? `CLAUDE.md`의 네이밍 규칙(`on[동작]`, `handle[동작]`, `is` 접두사 등)과 맞는가? `data`, `item`, `flag` 같은 모호한 이름이나 약어가 눈에 띄면 후보다.

**3. 단일 책임**
하나의 함수나 컴포넌트가 너무 많은 일을 하는가? 함수 길이보다 "이 함수를 한 문장으로 설명할 수 있는가"를 기준으로 판단한다. 설명에 "그리고"가 두 번 이상 들어가면 분리를 고려할 만하다.

**4. 불필요한 복잡도**
매직 넘버(설명 없이 등장하는 숫자·문자열), 3단계 이상 중첩된 조건문, 한 번만 쓰이는 추상화가 있는가? 코드를 처음 보는 사람이 멈추게 되는 지점이 어디인지 생각해본다.

**5. CLAUDE.md 컨벤션 불일치**
`export default` vs named export, Props 타입 선언 위치, 스타일 적용 방식이 프로젝트 패턴과 어긋나는가? 일관성 없는 패턴은 읽는 사람의 인지 부담을 높인다.

## [GATE] 개선 후보 보고 및 승인 대기

점검 결과를 아래 형식으로 정리해 개발자에게 제시한다.
개발자가 승인하거나 일부를 제외할 때까지 실제 코드 수정을 시작하지 않는다.

```
리팩토링 후보 목록

파일: src/hooks/useTagEditor.ts
  1. [네이밍] trimmed → normalizedValue — trim과 lowercase를 모두 수행하므로 'trimmed'만으로 의도 전달 불충분
  2. [중복] 없음

파일: src/components/TagInput.tsx
  1. [단일 책임] handleChange가 쉼표 감지와 onInputChange 호출을 동시에 담당 — 분리 검토 가능
  2. [네이밍] 없음

개선 불필요: src/types/note.ts, src/context/NotesContext.tsx
```

후보가 없으면: "리팩토링이 필요한 항목을 발견하지 못했습니다."라고 보고하고 종료한다.

---

## 5단계: 하나씩 변경 → 테스트 → 확인

승인된 항목을 **하나씩** 처리한다. 여러 개를 한꺼번에 수정하면 테스트가 깨졌을 때 원인을 추적하기 어렵다.

각 변경 후:

```bash
npm test
```

### 통과한 경우

다음 항목으로 이동한다.

### 실패한 경우

테스트가 깨지면 그 변경이 동작을 바꿨다는 신호다. 즉시 롤백하고 다른 접근을 시도하거나, 해당 항목을 건너뛴다.

```bash
git checkout -- <파일경로>
```

롤백 후에도 테스트가 실패하면 개발자에게 보고하고 중단한다.

---

## 6단계: 변경 요약

모든 항목 처리 후 아래 형식으로 요약한다.

```
리팩토링 결과

적용됨 (3건)
  useTagEditor.ts — trimmed → normalizedValue (네이밍)
  TagInput.tsx    — handleChange 분리: handleCommaInput, handleTextInput (단일 책임)
  NoteEditor.tsx  — 매직 문자열 상수화: PLACEHOLDER_TEXT (불필요한 복잡도)

건너뜀 (1건)
  TagInput.tsx — onAdd 인라인화 시도 → 테스트 실패로 롤백

최종 테스트: 24개 통과 / 0개 실패
```

---

## 제약

- **새 기능 추가 금지** — 리팩토링은 동작을 바꾸지 않는다. 새 기능이 필요하면 다음 이슈에서 다룬다.
- **테스트 파일 수정 금지** — 테스트는 리팩토링의 안전망이다. 테스트를 바꿔야 통과하는 리팩토링은 리팩토링이 아니다.
- **`src/` 밖 수정 금지** — 문서, 설정 파일은 이 단계의 범위가 아니다.
- **`git diff` 결과 밖 수정 금지** — 이번 이슈와 무관한 파일의 코드 품질은 별도 이슈에서 다룬다.
