# PRD 비교: 수동 vs my-prd 스킬

태그 기능 PRD를 수동으로 작성한 결과(`prd-manual.md`)와
`/my-prd` 스킬로 생성한 결과(`prd.md`)를 비교한다.

---

## 결론

**내용 품질은 동일하다.** ADR 수(4개), 결정 내용, 거부 이유, 트레이드오프가 일치한다.
차이는 형식 수준에 머물렀다.

---

## 항목별 비교

### 구조

| 항목      | 수동                                                  | 스킬 |
| --------- | ----------------------------------------------------- | ---- |
| 섹션 구성 | 개요 / 사용자 스토리 / ADR / Out of Scope / 용어 정의 | 동일 |
| ADR 수    | 4개                                                   | 4개  |
| ADR 주제  | 저장 위치 / 상태 분리 / 피드백 방식 / 최대 개수 UX    | 동일 |

### 형식 차이 (내용 차이 아님)

| 항목              | 수동                         | 스킬                         |
| ----------------- | ---------------------------- | ---------------------------- |
| Context 포맷      | `**Context**` 줄바꿈 후 본문 | `**Context** —` 인라인 한 줄 |
| Consequences 기호 | `(+)` / `(-)`                | `✅` / `⚠️`                  |
| 용어 정의 볼드    | 없음                         | 용어에 `**bold**` 적용       |

### 내용 차이 (미세)

| ADR                   | 수동                                                       | 스킬                                                                                                                  |
| --------------------- | ---------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| ADR-02 Context        | "어느 수준으로 관심사를 분리할지 결정이 필요하다"로 마무리 | "NoteEditor는 이미 3개 상태를 관리하고 있어 태그까지 추가하면 단일 책임이 흔들린다"를 추가 — 이유가 한 문장 더 구체적 |
| ADR-02 안 1 거부 이유 | "유효성 검사 테스트 시 NoteEditor 전체를 마운트해야 한다"  | "검증 로직 5가지를 테스트하려면" — 숫자를 명시해 더 구체적                                                            |
| ADR-02 안 2 거부 이유 | "로직 분리가 불완전하고"                                   | "어느 쪽도 책임이 불명확하고" — 표현 차이, 의미는 동일                                                                |

---

## 품질이 동일하게 나온 이유 분석

스킬 출력 품질이 수동과 같은 이유는 두 가지다.

**1. spec-fixed.md가 이미 기술 결정을 내포하고 있었다.**
태그의 spec-fixed.md는 데이터 구조(`tags: string[]`), 유효성 검사 5가지 규칙,
UI 동작, 엣지 케이스까지 모두 명세되어 있었다.
스킬이 "선택지를 탐색"하기보다 "이미 결정된 것을 ADR로 포장"하는 작업이었다.
spec-fixed.md 품질이 prd.md 품질을 결정한다.

**2. 스킬의 7가지 고정 기준이 빠뜨린 관점 없이 비교를 강제했다.**
수동 과정에서 자연스럽게 도달한 것과 동일한 비교 축(테스트 용이성, 기존 패턴 일관성 등)을
스킬이 구조적으로 보장했다.

---

## 스킬이 가져다준 실질적 차이

| 항목               | 수동                      | 스킬                                    |
| ------------------ | ------------------------- | --------------------------------------- |
| 아키텍처 비교 제시 | 대화 중 자연 발생         | 7가지 고정 기준 + 코드 예시로 구조화    |
| 추천 이유          | 요청 시 설명              | 비교 표 직후 2가지 이유로 자동 제시     |
| Out of Scope 도출  | spec-fixed.md 초안 그대로 | ADR 과정에서 새로 드러난 항목 추가 가능 |
| 재현 가능성        | 사람마다 다를 수 있음     | 동일한 흐름이 항상 보장됨               |

---

## 스킬 개선 포인트

1. **Context 포맷 통일** — 수동 스타일(`줄바꿈 후 본문`)이 단락이 길 때 더 읽기 쉽다.
   스킬 지시문에 포맷을 명시하면 일관성이 생긴다.
2. **Consequences 기호** — `✅` / `⚠️` vs `(+)` / `(-)` 중 하나로 고정하면 좋다.
3. **spec-fixed.md 품질 의존도** — 스킬이 spec의 완성도에 크게 의존한다.
   spec이 빈약하면 아키텍처 안이 추상적으로 나올 수 있다.
   spec 점검 단계를 스킬 시작 시 추가하는 것을 고려할 수 있다.
