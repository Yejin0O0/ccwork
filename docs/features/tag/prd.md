# 태그 기능 PRD

## 1. 개요

노트에 태그를 붙여 분류·맥락 정보를 추가할 수 있도록 한다.
태그는 노트 내 배열(`tags: string[]`)로 저장되며, `NoteEditor` 내에서 입력·수정·삭제가 가능하다.
이번 릴리스는 태그 입력 및 저장에 집중하며, 태그 기반 필터링은 포함하지 않는다.

---

## 2. 사용자 스토리

| #   | 역할       | 행동                                                    | 목적                                      |
| --- | ---------- | ------------------------------------------------------- | ----------------------------------------- |
| 1   | 사용자로서 | 노트 편집 화면에서 태그를 입력하고 싶다                 | 노트에 주제·카테고리 정보를 남기기 위해   |
| 2   | 사용자로서 | Enter 또는 쉼표(`,`)로 태그를 빠르게 추가하고 싶다      | 마우스 없이 키보드만으로 태그를 달기 위해 |
| 3   | 사용자로서 | 태그가 칩(badge) 형태로 보이길 원한다                   | 어떤 태그가 붙었는지 한눈에 파악하기 위해 |
| 4   | 사용자로서 | 잘못 추가한 태그를 삭제하고 싶다                        | 태그 목록을 정확하게 유지하기 위해        |
| 5   | 사용자로서 | 저장 버튼 하나로 제목·내용·태그를 한 번에 저장하고 싶다 | 저장 단계를 늘리지 않기 위해              |
| 6   | 사용자로서 | 유효하지 않은 태그 입력 시 즉각적인 피드백을 받고 싶다  | 어떤 입력이 거부되는지 바로 알기 위해     |

## 3. 기술 결정 (ADR)

### ADR-01. 태그 저장 위치

**Context** — 태그를 어디에, 어떤 구조로 저장할지 결정이 필요하다. 현재 앱은 JSON Server 기반 단일 `db.json`을 DB로 사용하며, 노트는 `/notes` 엔드포인트 하나로 관리된다. 태그는 노트와 1:N 관계이므로, 노트 문서 안에 포함시키거나 별도 컬렉션으로 분리하는 선택지가 있다.

**Decision** — `Note` 타입에 `tags: string[]` 필드를 추가해 노트 문서 안에 인라인으로 저장한다.

**Alternatives**

- **별도 `/tags` 컬렉션으로 분리**: 태그를 독립 엔티티로 정규화해 관리 → 거부. 현재 범위(태그 필터링·전체 태그 목록 없음)에서는 과도한 복잡도다. 노트 조회 시 태그를 별도로 fetch해야 하고, JSON Server 관계 설정이 추가로 필요해진다.

**Consequences**

- ✅ 노트 fetch 한 번으로 태그 포함 완전한 Note 객체 획득 가능
- ✅ `updateNote`가 이미 `Partial<Note>`를 받으므로 API·Context 레이어 변경 불필요
- ⚠️ 태그만 조회하거나 집계하는 기능(예: 태그별 노트 수)이 생기면 구조 재설계 필요
- ⚠️ 태그 데이터 무결성(중복, 허용 문자)을 DB 레벨에서 강제할 수 없어 클라이언트 검증에만 의존

---

### ADR-02. 상태·로직 분리 구조

**Context** — 태그 입력에는 유효성 검사 규칙 5가지(빈 값 무시, 최대 15자, 허용 문자, 중복, 최대 10개)와 소문자 정규화, shake 피드백이 얽혀 있다. 이 로직을 `NoteEditor`에 인라인으로 추가하면 기존 title/content 관리 코드와 뒤섞인다. NoteEditor는 이미 `title / content / saving` 3개 상태를 관리하고 있어 태그까지 추가하면 단일 책임이 흔들린다.

**Decision** — `useTagEditor` 커스텀 훅으로 태그 상태·로직 전체를 분리한다. `TagInput`은 훅 결과만 받아 렌더링하고, `NoteEditor`는 훅과 컴포넌트를 조합하는 역할만 한다.

```
useTagEditor(initialTags)  ← 상태·유효성 검사·정규화
TagInput(props)            ← 렌더링만
NoteEditor                 ← 훅 결과를 TagInput에 연결
```

**Alternatives**

- **안 1 — NoteEditor 인라인**: `tags`, `tagInput`, `shaking`을 NoteEditor 로컬 state로 추가, 유효성 검사도 인라인 → 거부. 이미 title/content/saving을 관리하는 NoteEditor에 태그 로직까지 추가하면 단일 책임 위반이 심화된다. 검증 로직 5가지를 테스트하려면 NoteEditor 전체를 마운트해야 한다.
- **안 2 — TagInput 컴포넌트만 분리**: NoteEditor가 `tags` state를 소유하고 TagInput에 props로 내려줌 → 거부. 검증 로직이 TagInput 내부 또는 NoteEditor에 분산되어 어느 쪽도 책임이 불명확하고, 훅 없이는 로직만 독립적으로 테스트하기 어렵다.

**Consequences**

- ✅ 유효성 검사·정규화·상태 전이를 `renderHook`으로 UI 마운트 없이 단위 테스트 가능
- ✅ `TagInput`이 순수 렌더 컴포넌트가 되어 스냅샷 테스트·스타일 작업이 단순해짐
- ✅ 커스텀 훅 기반 합성 패턴의 선례가 되어 다른 입력 컴포넌트에 재사용 가능
- ⚠️ 현재 코드베이스에 없던 커스텀 훅 패턴 도입 — 코드베이스 컨벤션으로 명시 필요
- ⚠️ `useTagEditor.ts`, `TagInput.tsx` 파일 2개 추가. 단순 기능 대비 파일 수 증가

### ADR-03. 유효성 검사 피드백 방식

**Context** — 태그 입력 시 허용 문자 위반, 중복, 최대 길이 초과 등 잘못된 입력이 발생할 수 있다. 사용자에게 "이 입력이 거부되었다"는 피드백을 전달할 방법이 필요하다. 현재 앱에는 toast·alert 컴포넌트가 없고, 폼은 모달 없이 인라인으로 표시된다.

**Decision** — toast·alert·에러 텍스트 없이 입력창 shake 애니메이션만으로 거부 피드백을 전달한다.

**Alternatives**

- **인풋 하단 에러 텍스트**: 오류 원인을 메시지로 구분 가능 → 거부. 에러 상태마다 문구를 관리해야 하고, 레이아웃 공간이 필요하다. 이번 범위의 작은 태그 입력 UI에 비해 무겁다.
- **toast 알림**: 눈에 잘 띔 → 거부. toast 컴포넌트 신규 도입이 필요하고, 폼 흐름 밖에서 알림이 뜨는 방식이 이 입력 패턴과 어울리지 않는다.
- **silent ignore**: 구현 최소 → 거부. 입력이 거부된 사실 자체를 사용자가 알 수 없어 혼란을 유발한다.

**Consequences**

- ✅ 추가 컴포넌트 없이 구현 가능. 폼 레이아웃 변경 없음
- ✅ 입력 실패를 즉각적으로 전달하면서 폼 흐름을 끊지 않음
- ⚠️ 허용 문자 위반인지, 중복인지, 길이 초과인지 원인을 구분할 수 없어 사용자가 이유를 파악하기 어려움
- ⚠️ shake 애니메이션은 시각에만 의존하는 피드백이라 접근성 측면에서 보완이 필요할 수 있음

### ADR-04. 최대 개수 제한 시 UX

**Context** — 태그는 노트당 최대 10개로 제한된다. 10개에 도달한 상태에서 추가를 시도할 때 처리 방식을 결정해야 한다. ADR-03에서 다른 유효성 오류는 shake로 처리하기로 했으나, 최대 개수 도달은 성격이 다르다. "이 입력이 잘못됨"이 아니라 "현재 입력 자체가 불가능한 상태"이기 때문이다.

**Decision** — 인풋을 `disabled` 처리하고 "태그는 최대 10개까지 추가할 수 있습니다" 안내 문구를 표시한다. 태그 삭제 후에는 인풋을 다시 활성화한다.

**Alternatives**

- **shake만 (ADR-03과 동일)**: 구현 일관성 높음 → 거부. 인풋이 활성화된 채로 계속 입력을 시도하면 shake만 반복해서 보게 된다. 사용자가 "왜 안 되는지" 알 수 없고, 10개 도달 상태가 명확히 전달되지 않는다.
- **silent ignore**: → 거부. 입력이 왜 안 되는지 전혀 알 수 없어 혼란을 유발한다.

**Consequences**

- ✅ 현재 상태(10개 도달)를 명확히 전달. 불필요한 입력 시도를 원천 차단
- ✅ 태그 삭제 후 자동 활성화되어 상태 전환이 직관적
- ⚠️ 다른 유효성 오류(shake)와 피드백 방식이 달라 일관성이 약간 저하됨
- ⚠️ disabled 스타일과 안내 문구 렌더링 조건 분기가 추가로 필요

---

## 4. Out of Scope

- 태그 기반 노트 필터링 / 검색
- 태그 자동완성 (기존 태그 목록 제안)
- 태그별 노트 수 집계 / 통계
- 태그 전용 관리 화면 (생성·수정·삭제)
- `NoteItem` 카드에서의 태그 표시
- 태그 순서 변경 (드래그 앤 드롭 등)

---

## 5. 용어 정의

| 용어                      | 정의                                                                                               |
| ------------------------- | -------------------------------------------------------------------------------------------------- |
| **태그(tag)**             | 노트에 붙이는 짧은 레이블 문자열. 소문자·한글·숫자·하이픈만 허용                                   |
| **칩(chip / badge)**      | 태그를 시각적으로 표현하는 UI 요소. 텍스트와 삭제 버튼으로 구성                                    |
| **정규화(normalization)** | 태그 입력을 소문자로 변환하는 처리 (`React` → `react`)                                             |
| **흔들림 피드백(shake)**  | 유효하지 않은 입력 시 인풋이 좌우로 흔들리는 애니메이션. toast·alert 없이 시각적으로만 에러를 전달 |
| **PATCH**                 | 노트의 일부 필드만 갱신하는 HTTP 메서드. 태그 저장 시 `updateNote`가 이를 사용                     |
