# 검색 기능 요구사항

## 개요

노트를 많이 쌓아둔 사용자가 특정 내용을 찾을 수 있도록, 제목과 본문을 대상으로 실시간 필터링 검색을 제공한다.

## 사용자 시나리오

**시나리오 1 — 본문으로 노트 찾기**
노트가 많이 쌓인 상태에서 특정 키워드가 포함된 노트를 찾고 싶다. 사이드바 상단 검색창에 키워드를 입력하면 제목·본문이 해당 키워드를 포함한 노트만 목록에 남는다.

**시나리오 2 — 검색 초기화**
검색어를 지우면 전체 노트 목록이 다시 표시된다.

**시나리오 3 — 검색 결과 없음**
어떤 노트에도 키워드가 없을 경우 "검색 결과가 없습니다" 메시지를 표시한다.

## 경계 조건 및 엣지 케이스

- 검색어가 빈 문자열이면 필터 없음으로 처리 → 전체 노트 표시
- 대소문자 구분 없이 검색한다 (case-insensitive)
- 검색 대상: `title` + `content` 두 필드
- debounce 없이 입력할 때마다 즉시 필터링
- 검색 중 노트를 선택하거나 편집하는 기존 동작은 그대로 유지된다

## 에러 처리 방식

- 검색 결과 0건: NoteList의 기존 빈 상태 패턴과 동일하게 "검색 결과가 없습니다" 메시지 표시
- 검색 자체는 클라이언트 필터링이므로 별도 에러 상태 불필요

## Out of Scope (초안)

- 검색어 하이라이팅 (매칭 텍스트 강조 표시)
- 검색 히스토리 저장
- 태그 / 즐겨찾기 복합 필터 (향후 확장 대상이지만 이번 이슈에서는 제외)
- 서버 사이드 검색 (JSON Server `_like` 파라미터 활용)
- 정규식 / 고급 검색 문법

## 용어 정의

| 용어                         | 정의                                                                              |
| ---------------------------- | --------------------------------------------------------------------------------- |
| **검색어 (query)**           | 사용자가 검색창에 입력한 문자열                                                   |
| **필터링 (filtering)**       | 클라이언트에서 notes 배열을 검색어로 좁히는 동작                                  |
| **검색 범위**                | `title` + `content` 두 필드                                                       |
| **빈 검색**                  | 검색어가 빈 문자열인 상태 → 전체 노트 표시                                        |
| **검색 상태 (search state)** | 현재 검색어를 담는 상태. 독립 훅으로 관리해 향후 복합 필터와 조합 가능하도록 설계 |
